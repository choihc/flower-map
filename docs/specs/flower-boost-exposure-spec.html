꽃 집중 노출(Boost) 기능 명세

특정 꽃을 기간 지정으로 우선 노출시키는 상위 그룹 고정 메커니즘
작성일 2026-06-22 최종 수정일 2026-06-22 flower-boost-exposure-spec.md

핵심 요약 (Executive Summary)

미디어·SNS에서 화제가 된 꽃이 자동 점수(now_score)에 밀려 노출되지 않는 문제를 해결한다. 어드민이 꽃 단위로 집중 노출 기간을 지정하면, 그 기간 동안 해당 꽃의 모든 공개 명소가 모바일 앱의 모든 노출 지점에서 상위 그룹으로 고정된다.

  • 방식 상위 그룹 고정 — 부스트 그룹을 위로, 그룹 내부는 기존 정렬(점수/순서) 유지
  • 범위 모바일 앱 — 홈 TOP · 검색 · 명소목록
  • 운영 기간 지정 + 자동 만료 — 종료일 경과 시 조작 없이 해제(배치 불필요)
  • 강도 단일 on/off — 꽃 간 우선순위 구분 없음
  • 데이터 flowers 테이블에 boost_start_at · boost_end_at 두 컬럼 추가

1. 개요

1.1 정의

집중 노출(Boost)은 어드민이 특정 꽃(flower)에 대해 기간을 지정하여, 해당 기간 동안 그 꽃에 속한 명소(spot)들을 다른 꽃의 명소보다 항상 상위에 노출시키는 기능이다. 기존 자동 점수(now_score)·수동 정렬(display_order)과 독립적으로 동작하는 상위 그룹 고정(pinning) 메커니즘이다.

1.2 도메인 책임

책임설명
집중 노출 설정어드민이 꽃 단위로 시작일/종료일을 지정·해제
활성 부스트 판정조회 시점(오늘)을 기준으로 "현재 집중 노출 중"인지 판정
상위 그룹 고정활성 부스트 꽃의 명소를 모든 노출 지점에서 비부스트보다 상위 배치
자동 만료종료일 경과 시 별도 조작 없이 자동 해제

1.3 해결하는 문제

특정 꽃이 미디어·SNS에서 화제가 되어 노출 수요가 급증했으나, 자동 점수 산정상 다른 꽃에 밀려 충분히 노출되지 않는 상황에서, 운영자가 전략적·일시적으로 노출 빈도를 끌어올릴 수 있게 한다.

2. 용어

용어정의
부스트 기간boost_start_at ~ boost_end_at (양 끝 포함, 일 단위)
활성 active boost조회 시점이 부스트 기간 안에 들어 실제 우선순위가 적용되는 상태
예약 scheduled기간이 설정됐으나 시작일이 아직 도래하지 않은 상태
만료 expired종료일이 지나 비활성화된 상태
부스트 그룹한 화면에서 활성 부스트 꽃에 속한 명소 집합 (상단 배치)

3. 데이터 모델

3.1 flowers 테이블 확장

ALTER TABLE public.flowers
  ADD COLUMN boost_start_at date,   -- 집중 노출 시작일 (nullable)
  ADD COLUMN boost_end_at   date;   -- 집중 노출 종료일 (nullable)
  • 기본값 둘 다 NULL(미설정)
  • 별도 boolean 플래그를 두지 않는다. 활성 여부는 항상 두 날짜와 조회 시점으로 도출 (단일 진실 원천 = 기간)
  • 기존 sort_order, is_active는 의미 변경 없이 유지, 집중 노출과 독립 동작

3.2 활성 부스트 판정 규칙

isBoosted(f, today) :=
  f.boost_start_at IS NOT NULL
  AND f.boost_end_at IS NOT NULL
  AND f.boost_start_at <= today
  AND today <= f.boost_end_at
핵심: 별도 배치(cron) 없이 조회/매핑 시점의 날짜 비교만으로 활성/예약/만료를 판정한다. 시작일이 미래면 예약, 종료일이 과거면 만료.

4. 기능 요구사항 (FR)

FR-1 어드민 — 집중 노출 설정

  • FR-1-1 꽃 편집 화면(FlowerForm)에서 시작일·종료일 입력
  • FR-1-2 두 날짜를 비우고 저장 시 해제(둘 다 NULL)
  • FR-1-3 기존 updateFlower 경로로 저장, 저장 후 공개 노출 경로 재검증(revalidate)

FR-2 어드민 — 상태 표시·검증

  • FR-2-1 꽃 목록에 상태 배지: 집중 노출 중(D-n) 예약(MM.DD~) 만료 (없음)
  • FR-2-2 배지 구분은 FR-3 판정 규칙과 동일 기준
  • FR-2-3 검증: ① 둘 다 입력하거나 둘 다 비움(한쪽만 불가) ② start <= end. 위반 시 저장 거부

FR-3 활성 부스트 판정

  • FR-3-1 §3.2 isBoosted 규칙으로 판정
  • FR-3-2 기준 날짜는 KST "오늘"

FR-4 노출 정렬 — 상위 그룹 고정 (공통 계약)

  • FR-4-1 모든 노출 지점에서 부스트 그룹이 비부스트보다 항상 상위
  • FR-4-2 부스트 그룹 내부 순서는 화면의 기존 기준(now_score/display_order/종료 임박순) 유지
  • FR-4-3 비부스트 그룹 내부 순서도 기존 기준 유지
  • FR-4-4 여러 꽃 동시 부스트 시 꽃 간 구분 없음(단일 on/off), 그룹 전체를 기존 기준으로 정렬
  • FR-4-5 안정 정렬 — 1차 키 부스트 여부, 2차 키 기존 기준과 동치

FR-5 적용 지점별 동작 모바일 앱

지점동작
FR-5-1 모바일 홈 TOPgetTopSpots(n)에서 부스트 명소가 limit n에 의해 누락되지 않도록 우선 포함. 부스트 먼저 채우고 남은 자리를 now_score 상위로 채운 뒤 FR-4로 정렬한 결과와 동치
FR-5-2 모바일 검색 추천순compareByRecommendation이 1차 부스트, 2차 now_score desc
FR-5-3 모바일 명소목록기본(display_order) 및 "종료 임박순" 모두 부스트 그룹 상위. 내부는 각 모드 기준

FR-6 자동 만료

  • FR-6-1 종료일 경과 꽃은 다음 조회부터 비부스트 처리
  • FR-6-2 만료 설정 데이터는 자동 삭제하지 않음(이력·재설정 보존)

5. 비기능 요구사항 (NFR)

ID요구사항
NFR-1모바일 앱 격리 구현 — 모바일은 자체 spotRepository/spotMappers로 정렬 계약(FR-4)을 구현한다. 공유 패키지(flower-domain/supabase)와 독립적으로 동작한다.
NFR-2공유 패키지 영향 범위 — flower-domain·supabase는 boost 기능에서 사용하지 않는다(모바일/어드민 자체 코드 사용). flowers 테이블 스키마 변경만 DB 차원의 공통 사항이다.
NFR-3네트워크 최소화 — 부스트 정보는 기존 flower 조인 select에 컬럼 추가. 목록은 추가 쿼리 없이 메모리 정렬. 홈 TOP만 보조 쿼리 1회 허용
NFR-4배치 불필요 — 만료 전이는 조회 시점 날짜 비교로 도출
NFR-5시간대 일관성 — "오늘"은 KST 기준, 기존 개화 D-day와 동일 경계
NFR-6기존 동작 보존 — 미설정 상태에서 모든 정렬 결과가 현행과 동일(회귀 없음)
NFR-7어드민 권한 — 기존 admin_users 체계 내, 신규 권한 추가 없음

8. 외부 의존성

  • Supabase PostgreSQL — flowers 스키마 변경(마이그레이션), flower 조인 컬럼 확장
  • 기존 어드민 인증 — admin_users 기반(변경 없음)
  • 신규 외부 API·라이브러리 없음

9. 명시적 가정

  • 조정 단위는 꽃(명소 단위 is_featured와 독립). 한 꽃 부스트 시 그 꽃의 모든 공개 명소 포함
  • 집중 노출은 공개 여부(status='published')를 바꾸지 않음. 비공개 명소는 대상 아님
  • 개화 종료 여부와 무관하게 적용(운영자 의도 우선)
  • Supabase가 date를 'YYYY-MM-DD' 문자열로 반환(기존 bloom_start_at과 동일)

10. 범위 외 (Out of Scope)

  • 꽃 간 부스트 우선순위(가중치·레벨) — 단일 on/off로 확정
  • 명소 단위 개별 부스트
  • 부스트 성과(클릭·전환) 측정·리포팅
  • 부스트 설정 변경 이력(audit log)
  • 푸시 알림·홈 큐레이션 연동
  • 토스 미니앱: 프로젝트가 삭제되어 본 기능의 적용 대상이 아니다.
