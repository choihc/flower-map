모바일 홈 로딩 동작 스펙

대상: apps/mobile (Expo) P1 점진적 렌더링 P2 캐시 영속화
작성일 2026-06-19 최종 수정일 2026-06-19 관련 플랜 docs/plans/mobile-home-loading.md (작성 예정)
대체 관계 — 본 스펙은 2026-06-05-home-loading-skeleton-spec.md 의 "통합 대기 게이트" 모델(FR-1~FR-5, useHomeReady)을 대체(supersede)한다. SkeletonBox 공용 프리미티브 정의(구 스펙 5.1)는 변경 없이 계승한다.

핵심 요약

현재 홈은 4개 쿼리가 모두 끝날 때까지(또는 5초 타임아웃) 화면 전체를 단일 스켈레톤으로 막는다(useHomeReady 게이트). 가장 느린 쿼리 하나에 빠른 섹션까지 묶인다. 또 영속 캐시가 없어 앱을 켤 때마다 네트워크 왕복 후에야 콘텐츠가 뜬다.

본 스펙은 두 가지로 해결한다.

  • P1 점진적 렌더링 — 전체 게이트 제거. 셸은 즉시, 각 섹션은 자기 데이터가 준비되는 즉시 독립 노출.
  • P2 캐시 영속화 — React Query 캐시를 AsyncStorage에 영속(24h). 재방문 시 직전 데이터 즉시 표시 후 백그라운드 갱신.

데이터는 서버 cron으로 하루 1~2회만 갱신되므로 공격적 캐싱이 신선도를 해치지 않는다.

1개요

모바일 앱(apps/mobile, Expo) 홈 화면(HomeScreen)의 데이터 로딩·렌더링 동작과, 이를 뒷받침하는 앱 전역 React Query 캐시 영속화의 명세다. apps/web·apps/toss-mini·공유 패키지와 무관하다.

결과물정의
P1 점진적 렌더링화면 셸은 즉시 표시, 각 섹션은 자기 데이터 준비 즉시 독립 노출. 화면 전체를 단일 게이트로 묶지 않음.
P2 캐시 영속화React Query 캐시를 디바이스 저장소에 영속. 콜드 스타트 시 직전 데이터 즉시 표시 + 백그라운드 갱신(stale-while-revalidate).

2기능 요구사항 (FR)

2.1 점진적 렌더링 P1

ID요구사항
FR-1화면 셸 즉시 표시 — 진입 시 ScreenShell(헤더·레이아웃·스크롤)은 데이터를 기다리지 않고 즉시 렌더.
FR-2전체 대기 게이트 부재 — 통합 준비 판정(useHomeReady)·전체 화면 통합 스켈레톤(HomeSkeleton)은 존재하지 않음. 한 섹션의 pending이 다른 섹션을 막지 않음.
FR-3섹션 독립 렌더링 — 각 섹션은 자기 쿼리 상태에 따라 독립 렌더(아래 매핑).
FR-4섹션별 로딩 표시 — 의존 쿼리 pending 동안 자기 영역 크기의 SkeletonBox 스켈레톤 표시. 셸·헤더는 즉시 노출.
FR-5빈 상태 — success+데이터 없음이면 섹션 숨김/안내. 다른 섹션에 영향 없음.
FR-6에러 상태 — error면 해당 섹션만 빈/에러 처리(숨김 허용) + console.error 관측. 다른 섹션 정상.

2.2 쿼리 캐시 영속화 P2

ID요구사항
FR-7캐시 영속 — 전역 캐시를 AsyncStorage에 영속, 콜드 스타트 시 복원.
FR-8즉시 표시 후 갱신 — 복원 캐시 있으면 즉시 표시(무스켈레톤) + 백그라운드 최신화(stale-while-revalidate).
FR-9캐시 만료 — 보존 기한 24h(maxAge). 초과 시 복원 안 함.
FR-10캐시 무효화 — 앱 버전/스키마 버전 변경 시 기존 캐시 폐기(buster).
FR-11영속 대상 — success 쿼리만 영속. 에러/pending 제외.

3비기능 요구사항 (NFR)

ID요구사항
NFR-1앱 격리 — 변경은 apps/mobile/에 국한. 타 앱·공유 패키지 동작 불변.
NFR-2신선도 — 일 단위 갱신에 맞춘 staleTime/maxAge. 최대 24h 캐시 + 진입 시 갱신.
NFR-3테스트 — 신규·변경 동작 모두 자동 테스트. SkeletonBox 애니메이션이 테스트 렌더를 깨지 않음.
NFR-4성능 — 유효 캐시 재방문 시 네트워크 왕복 없이 첫 콘텐츠 표시.
NFR-5의존성 최소 — 공식 TanStack persist 패키지 + 기존 AsyncStorage만. 그 외 신규 의존성 금지.
NFR-6중복 요청 없음 — 동일 키 공유 섹션 간 중복 네트워크 요청 없음.

5구성 요소별 책임·계약

5.1 queryClient (수정)

export const queryClient = new QueryClient({
  defaultOptions: {
    queries: {
      staleTime: 1000 * 60 * 30,      // 30분(데이터 일 단위 갱신 — 상향)
      gcTime: 1000 * 60 * 60 * 24,    // 24h — persist maxAge 이상이어야 복원 대상 유지
      retry: 2,
    },
  },
});

영속 복원이 동작하려면 각 쿼리의 gcTime이 maxAge 이상이어야 한다. 기존 기본 gcTime(5분)은 부적합하므로 상향한다.

5.2 queryPersister (신규)

import { createAsyncStoragePersister } from '@tanstack/query-async-storage-persister';
import AsyncStorage from '@react-native-async-storage/async-storage';

export const CACHE_BUSTER: string;                 // 앱 버전 또는 캐시 스키마 버전
export const PERSIST_MAX_AGE = 1000 * 60 * 60 * 24; // 24h (FR-9)

export const asyncStoragePersister = createAsyncStoragePersister({
  storage: AsyncStorage,
  key: 'flower-map-rq-cache',
});

export const persistOptions = {
  persister: asyncStoragePersister,
  maxAge: PERSIST_MAX_AGE,
  buster: CACHE_BUSTER,
  dehydrateOptions: {
    shouldDehydrateQuery: (query) => query.state.status === 'success', // FR-11
  },
};

5.3 _layout.tsx (수정)

import { PersistQueryClientProvider } from '@tanstack/react-query-persist-client';

<PersistQueryClientProvider client={queryClient} persistOptions={persistOptions}>
  <AuthProvider> ... </AuthProvider>
</PersistQueryClientProvider>

  • useHomeReady 호출·ready 분기·HomeSkeleton 반환 제거.
  • ScreenShell 내부에 각 섹션 직접 합성. 각 섹션이 자기 쿼리·로딩/빈/에러 처리.

상태동작

8명시적 가정

  • 조회 함수·쿼리 키는 현행 그대로 존재하며, 섹션들이 동일 키로 캐시를 공유한다(NFR-6).
  • React Query v5의 status 의미(pending=최초 응답 전, 에러 시 pending 해제)를 따른다.
  • react-query-persist-client는 복원을 비동기 수행하며, 복원 완료 전에는 섹션이 스켈레톤을 보일 수 있다(허용). 완료 시 즉시 데이터로 전환.
  • AsyncStorage는 테스트에서 mock 대체 가능하며, 영속 설정 검증은 설정 객체 계약(maxAge·buster·dehydrate 조건) 단위로 가능하다.
  • CACHE_BUSTER는 앱 버전(app.config.ts의 version) 또는 별도 캐시 스키마 버전에서 파생(플랜에서 확정).
