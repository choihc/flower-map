# 앱인토스용 `꽃 어디` 미니앱 설계

## 1. 목표

기존 `꽃 어디` 모바일 앱과 배포 흐름에 영향을 주지 않으면서, 앱인토스용 미니앱을 별도 워크스페이스에서 개발한다.

이번 설계의 핵심 목표는 아래와 같다.

- 기존 `apps/mobile` 안드로이드 승인 및 배포 일정에 영향 주지 않기
- 앱인토스 전용 UI를 별도로 구현하되 조회 로직은 재사용 가능하게 설계하기
- 로그인 없이도 사용할 수 있는 읽기 중심 MVP를 빠르게 만들기
- `네이버지도 렌더링 + 마커 + 상세 진입`을 P0 핵심 기능으로 다루기

## 2. 범위

### 포함

- 앱인토스 React Native 미니앱 스캐폴드 구성
- 홈 화면
- 꽃/지역 기반 명소 리스트
- 네이버지도 렌더링
- 지도 마커 표시 및 마커 선택
- 현재 위치 기반 근처 명소 탐색
- 명소 상세 화면
- 로컬 저장 기반 즐겨찾기/좋아요
- 외부 지도 앱 또는 길찾기 액션

### 제외

- 로그인
- 계정 기반 동기화
- 리뷰/커뮤니티
- 사용자 업로드
- 기존 Expo 앱 구조 리팩터링

## 3. 최상위 전략

앱인토스용 개발은 메인 워크스페이스와 분리된 별도 작업 공간에서 진행한다. 현재 기준으로는 `.worktrees/toss-mini`를 앱인토스 전용 작업 공간으로 사용한다.

핵심 전략은 다음과 같다.

- 앱인토스 앱은 `apps/toss-mini`에서 독립적으로 개발한다.
- 기존 `apps/mobile`은 앱인토스 작업 때문에 수정하지 않는다.
- 공용 가능한 것은 UI가 아니라 도메인/조회 로직까지만 분리한다.
- 저장/좋아요는 인터페이스만 공용화하고 구현은 플랫폼별로 분리한다.

## 4. 워크스페이스 구조

권장 구조는 아래와 같다.

```text
.worktrees/toss-mini/
  apps/
    toss-mini/
      granite.config.ts
      package.json
      src/
        app/
        screens/
        features/
        ui/
        lib/
  packages/
    flower-domain/
      src/
        types/
        queries/
        mappers/
        services/
    storage-adapters/
      src/
        favorites/
    supabase/
```

### 역할

#### `apps/toss-mini`

- 앱인토스 전용 진입점
- Granite 설정
- Toss/TDS 기반 화면 구성
- 지도 화면, 마커 인터랙션, 상세 화면
- 앱인토스 Storage 연동

#### `packages/flower-domain`

- 꽃/명소 타입 정의
- 목록/상세 조회용 함수
- Supabase 응답을 UI 친화 모델로 변환하는 매퍼
- 홈/지도/상세에서 공통으로 쓰는 필터 및 정렬 로직

#### `packages/storage-adapters`

- 즐겨찾기 저장 인터페이스
- 앱인토스 로컬 저장 구현체
- 추후 Expo 앱용 구현체를 붙일 수 있는 확장 지점

#### `packages/supabase`

- 공용 Supabase 클라이언트 및 타입
- 이미 존재하는 패키지가 있다면 유지하고, 부족한 부분만 최소 추가

## 5. 아키텍처 원칙

### 5.1 기존 앱 보호

- `apps/mobile`의 화면, 라우팅, 네이버지도 연동 코드는 이번 작업에서 직접 수정하지 않는다.
- 앱인토스용 개발 편의를 위해 기존 Expo 앱 코드를 복사해 시작하지 않는다.
- 메인 앱과 앱인토스 앱 사이의 공유는 `packages/*`를 통해서만 이뤄지게 한다.

### 5.2 공용화 범위

공용화 가능한 항목:

- `Spot`, `Flower`, `SpotDetail` 등 도메인 타입
- 명소 목록 조회
- 명소 상세 조회
- 홈 큐레이션용 조회
- 필터 파라미터 정규화
- 정렬/매핑/가공 로직

공용화하지 않을 항목:

- 화면 컴포넌트
- 라우팅
- 상태관리 훅
- 네이버지도 UI
- 저장 버튼/카드/상세 레이아웃
- 플랫폼 SDK 연동 코드

### 5.3 저장소 추상화

즐겨찾기/좋아요는 같은 기능명 아래에서 구현체를 분리한다.

예시 인터페이스:

```ts
export interface FavoriteStore {
  getAll(): Promise<string[]>;
  has(spotId: string): Promise<boolean>;
  toggle(spotId: string): Promise<boolean>;
}
```

앱인토스에서는 공식 `Storage` API를 사용한 로컬 저장 구현체를 둔다. 이 방식은 로그인 없이도 안정적으로 동작하며, MVP에 필요한 즐겨찾기 경험을 제공할 수 있다.

## 6. 기능 설계

### 6.1 홈

- 시즌 꽃 빠른 진입
- 추천 명소 섹션
- 현재 위치 기반 근처 명소 CTA
- 저장한 명소로 이동하는 진입점

### 6.2 지도

- 네이버지도 렌더링
- 명소 마커 표시
- 선택된 마커 강조
- 마커 선택 시 요약 카드 표시
- 카드 탭 시 상세 화면 이동
- 현재 위치 표시 또는 현재 위치 기준 탐색

### 6.3 리스트

- 꽃 종류 필터
- 지역 필터
- 거리/추천 기준 정렬
- 지도와 연계된 동일 데이터셋 사용

### 6.4 상세

- 대표 이미지
- 명소명, 꽃 종류, 시즌/행사 기간
- 설명, 주소, 운영 정보
- 즐겨찾기 토글
- 길찾기 액션

### 6.5 저장

- 로그인 없는 로컬 저장형 즐겨찾기
- 저장 목록 화면 제공
- 상세/리스트/지도 상태와 연동

## 7. 지도 기능을 P0로 두는 이유

사용자 요구와 서비스 본질상, 앱인토스용 `꽃 어디`는 단순한 목록 앱이 아니라 지도 기반 탐색 경험이 핵심이다. 따라서 이번 MVP에서 지도는 옵션이 아니라 P0다.

P0 지도 요구사항:

- 네이버지도 뷰 렌더링
- 마커 렌더링
- 마커 탭 이벤트 수신
- 선택된 명소 요약 정보 노출
- 상세 화면 전환
- 현재 위치 기반 초기 탐색

## 8. 지도 연동 기술 검증 전략

앱인토스 문서상 React Native 미니앱과 위치 권한은 공식 지원이 확인되지만, 현재 시점에는 `@mj-studio/react-native-naver-map`이 Granite 환경에서 문제 없이 동작한다는 명시 문구까지는 확인하지 못했다.

따라서 설계상 네이버지도는 P0로 두되, 구현 순서의 첫 단계는 아래 기술 검증 스파이크로 잡는다.

### 검증 목표

- Granite 기반 앱인토스 RN 프로젝트에서 네이버지도 네이티브 모듈 설치 가능 여부 확인
- 지도 뷰가 실제 렌더링되는지 확인
- 마커 렌더링 및 탭 이벤트 수신 가능 여부 확인
- 현재 위치 권한과 지도 카메라 이동 흐름 확인

### 성공 조건

- 지도 화면 최초 렌더링 성공
- 임의 좌표 기준 마커 1개 이상 렌더링 성공
- 마커 탭 시 JS 레벨 이벤트 수신 성공

### 실패 시 대체안

- 1차 릴리즈는 리스트 중심 + 외부 길찾기 UX로 축소
- 지도 기능은 별도 트랙으로 유지
- 다만 제품 설계상 지도는 계속 핵심 기능으로 남긴다

## 9. 데이터 흐름

```text
Supabase
  -> flower-domain queries
  -> mapper/service layer
  -> toss-mini screen models
  -> UI rendering

AppsInToss Storage
  -> favorite adapter
  -> screen state composition
  -> liked/saved UI 반영
```

목록, 지도, 상세는 동일한 도메인 조회 계층을 사용한다. 즐겨찾기는 별도 로컬 저장 계층에서 `spotId` 집합을 읽어와 화면 데이터에 합성한다.

## 10. 권한 및 플랫폼 연동

현재 설계에서 필요한 앱인토스 권한은 최소 아래 수준으로 본다.

- `geolocation: access`

추가 권한은 실제 기능 필요성이 확인될 때만 넣는다. MVP에서 불필요한 권한 확장은 피한다.

## 11. 구현 순서 제안

1. `apps/toss-mini` Granite 앱 기본 구조 정리
2. 네이버지도 기술 검증 스파이크 구현
3. `flower-domain` 조회 계층 정리
4. 홈/지도/상세 화면 기본 골격 구현
5. 로컬 저장형 즐겨찾기 구현
6. 길찾기 및 기본 UX 마감

## 12. 검증 계획

- 타입체크 통과
- Granite 설정 검증
- 지도 렌더링 수동 검증
- 마커 탭 인터랙션 검증
- 즐겨찾기 저장/해제 수동 검증
- 기존 `apps/mobile` 무수정 상태 확인

## 13. 의사결정 요약

- 앱인토스 개발은 별도 워크스페이스에서 진행한다.
- 기존 Expo 앱에는 영향 주지 않는다.
- 공용화는 조회 로직과 타입까지만 한다.
- 로그인은 넣지 않는다.
- 저장/좋아요는 앱인토스 로컬 저장 기반으로 구현한다.
- 네이버지도 연동은 P0이며, 가장 먼저 기술 검증한다.
