# 웹 플랫폼 통합 전환 설계

## 1. 목표

기존 `apps/web`과 `apps/toss-mini`의 역할을 재정의해서, 공개 사용자 경험은 Next.js 웹으로 통합하고 관리자 기능은 `/admin` 경로로 분리한다.

핵심 목표:
- 공개 사용자용 서비스와 관리자용 서비스를 하나의 Next.js 앱에서 운영한다.
- 토스 인앱 웹뷰 유입 화면은 TDS 기반 웹 UI로 구현한다.
- 지도 기능은 RN 네이티브 지도 대신 네이버 지도 웹 SDK로 통일한다.
- 초기 공개 서비스와 토스 인앱 서비스는 비로그인 읽기 중심으로 운영한다.

## 2. 범위

### 포함
- `apps/web`을 메인 웹 앱으로 재정의
- 공개 사용자 라우트 설계 및 구현 준비
- 기존 관리자 화면을 `/admin/*` 이하로 정리
- 토스 인앱 웹뷰 친화 UI 기준 수립
- 비로그인 기준 저장/탐색 흐름 정리

### 제외
- 공개 사용자 로그인
- 서버 동기화형 즐겨찾기
- 토스 외 제3자 소셜 로그인
- 네이티브 RN 기반 사용자 화면 유지

## 3. 제품 구조

### 3.1 앱 역할 재정의

- `apps/web`
  - 공개 사용자 웹 + 관리자 웹을 함께 제공하는 메인 Next.js 앱
- `apps/toss-mini`
  - 신규 사용자 기능 개발 중심이 아니라, 기존 레퍼런스 및 토스 연동 검증용 기준점
- `apps/mobile`
  - 별도 네이티브 앱 전략이 필요할 때만 유지, 현재 전환 범위의 중심은 아님

### 3.2 라우트 구조

공개 서비스와 관리자 서비스를 경로 기준으로 명확히 분리한다.

```text
/
/map
/search
/spot/[slug]
/support
/privacy

/admin
/admin/flowers
/admin/spots
/admin/spots/import
/admin/settings
```

현재 보호 경로가 `/`, `/settings`, `/flowers`, `/spots` 중심으로 잡혀 있으므로, 향후 미들웨어 보호 범위도 `/admin` prefix 중심으로 재정의해야 한다.

## 4. 공개 웹 설계

### 4.1 UX 원칙

- 공개 웹은 일반 브라우저에서도 자연스럽게 동작해야 한다.
- 동시에 토스 인앱 웹뷰에서 바로 사용할 수 있도록 TDS 기반 UI와 모바일 우선 정보 구조를 따른다.
- 즉, “마케팅 사이트 + 관리자 툴”이 아니라 “토스 인앱 진입도 감당하는 사용자 서비스 웹”으로 설계한다.

### 4.2 공개 라우트 역할

- `/`
  - 홈, 추천 명소, 시즌 진입점
- `/map`
  - 네이버 지도 웹 SDK 기반 지도 탐색
- `/search`
  - 텍스트 검색 + 꽃/지역 필터
- `/spot/[slug]`
  - 명소 상세, 사진, 길찾기, 저장 액션

### 4.3 공개 웹의 인증 정책

- 초기에는 비로그인 서비스로 운영한다.
- 저장 기능은 우선 로컬 저장으로 제한한다.
- 토스 인앱 환경에서는 필요 시 Apps-in-Toss Storage를 연결하고, 일반 웹에서는 브라우저 저장소 기반으로 분기할 수 있다.
- 서버 기반 계정 체계는 후속 단계에서 도입한다.

## 5. 토스 인앱 웹뷰 설계

### 5.1 전제

- 토스 인앱은 단순 일반 웹이 아니라 앱인토스 WebView 앱 환경이다.
- 따라서 공개 사용자 핵심 화면은 TDS 기반으로 구현해야 한다.
- RN 버전에서 사용하던 `@toss/tds-react-native`의 역할을 웹 환경에서는 TDS Web 계열 컴포넌트가 대체한다.

### 5.2 유지할 제약

- 모바일 우선 레이아웃
- 큰 터치 영역
- 토스 인앱 문맥에 맞는 상호작용
- 호스트 권한/저장소 연동이 필요한 경우 Apps-in-Toss 웹 프레임워크 활용

### 5.3 토스 인앱에서의 기능 정책

- 로그인 없음
- 읽기 중심 UX
- 저장은 로컬 저장만 우선 지원
- 외부 길찾기 링크 허용

## 6. 지도 전략

### 6.1 변경 방향

- RN 네이티브 지도(`@mj-studio/react-native-naver-map`) 의존을 제거한다.
- 공개 웹과 토스 인앱 모두에서 사용할 수 있도록 네이버 지도 웹 SDK 기반으로 통일한다.

### 6.2 영향

- 지도 화면의 렌더링 방식이 전면 교체된다.
- 마커 선택, 카드 동기화, 현재 위치 기반 이동, 상세 진입 동선은 웹 기준으로 다시 설계한다.
- 기존 RN 기반 fallback 메시지는 제거 대상이다.

## 7. 관리자 웹 설계

### 7.1 구조

- 기존 `(dashboard)` 성격의 관리자 페이지를 `/admin` 하위로 이동한다.
- 공개 사용자 레이아웃과 관리자 레이아웃은 분리한다.
- 관리자 인증/권한 검사는 `/admin` 범위에만 적용한다.

### 7.2 기대 효과

- 공개 루트(`/`)를 로그인 없이 열 수 있다.
- 관리자 보호 경계가 경로 기준으로 단순해진다.
- 배포는 단일 앱으로 유지하면서도 정보구조는 명확해진다.

## 8. 기술 구조

### 8.1 유지

- Next.js App Router
- Supabase 데이터 소스
- 기존 관리자용 서버 액션/데이터 접근 코드

### 8.2 신규 또는 변경

- 공개 사용자용 route group 또는 public layout
- `/admin` 기준 middleware 재정의
- 네이버 지도 웹 SDK 래퍼 컴포넌트
- TDS 기반 공개 사용자 UI 컴포넌트
- 저장소 어댑터 추상화

## 9. 리스크와 대응

### 9.1 공개 웹과 관리자 웹 공존 리스크

- 하나의 앱에서 서로 다른 UI 톤을 다뤄야 한다.
- 대응:
  - 레이아웃과 라우트 그룹을 분리한다.
  - 관리자 UI와 공개 UI는 컴포넌트 경계를 분리한다.

### 9.2 토스 인앱/일반 웹 저장소 차이

- 토스 인앱은 Apps-in-Toss Storage, 일반 웹은 브라우저 저장소를 쓰게 될 수 있다.
- 대응:
  - 저장소 인터페이스를 먼저 추상화한다.

### 9.3 지도 SDK 전환 리스크

- 기존 RN 지도 상호작용을 그대로 옮길 수는 없다.
- 대응:
  - 지도는 독립 작업 단위로 분리하고, 화면 단위 프로토타입부터 검증한다.

## 10. 결론

현재 조건에서는 `apps/web`을 메인 웹 플랫폼으로 승격하고, 공개 서비스는 비로그인 TDS 기반 웹으로 확장하며, 관리자 기능은 `/admin`으로 재배치하는 방향이 가장 효율적이다.

토스 인앱은 별도 RN 앱을 계속 키우기보다, WebView 앱 제약을 만족하는 웹 구현을 중심으로 가져가는 것이 유지보수와 배포 모두에서 유리하다.
